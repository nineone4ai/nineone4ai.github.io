# 设计评审门禁

## 定义

设计评审门禁（Design Review Gate）是在"需求交底完成"与"编码开始"之间插入的一个**强制性对齐检查点**。

只有当设计产物通过多角色联合评审并签确后，服务才可以进入编码阶段。它是解决"需求理解不一致"这一根因问题最高效的干预手段。

## 为什么需要它

在典型的微服务研发流程中，断层最常发生在：

```
需求交底 ✅ → [空白地带] → 开发自行理解 → 编码 → 联调时爆发冲突 ❌
```

设计评审门禁将"空白地带"替换为：

```
需求交底 ✅ → 设计产物输出 → 多角色联合评审 → 签确通过 → 编码 ✅
```

## 触发条件

每个服务在进入编码阶段前，必须满足以下条件才能通过门禁：

### 必须产出（缺一不可）

- [ ] **接口文档**：OpenAPI 3 / Swagger 规范，包含请求参数、响应结构、状态码、错误码
- [ ] **数据库设计**：ER图 + 建表SQL，字段类型、索引、分区策略说明
- [ ] **关键链路时序图**：覆盖核心业务场景的服务间调用时序
- [ ] **异常流程设计**：降级策略、重试逻辑、超时处理、分布式事务边界

### 评审 Checklist（技术维度）

**接口设计:**
- [ ] 接口命名符合 RESTful 规范
- [ ] 所有入参有合法性校验说明
- [ ] 分页接口有 pageSize 上限控制
- [ ] 幂等性设计（POST/PUT 操作需说明幂等方案）

**数据库设计:**
- [ ] 主键策略（雪花ID/自增/UUID）已确认
- [ ] 敏感字段加密方案说明
- [ ] 大表分页查询无全表扫描风险
- [ ] 索引覆盖主要查询场景

**分布式设计:**
- [ ] 服务间调用超时/熔断配置（Sentinel）
- [ ] 分布式事务边界清晰（是否需要Seata/消息补偿）
- [ ] 缓存穿透/击穿/雪崩防护方案

## 参与角色

| 角色 | 职责 |
|------|------|
| 产品经理 | 确认设计符合需求意图，验收用例与接口对应 |
| 开发负责人 | 技术方案可行性、与现有架构一致性 |
| QA负责人 | 测试用例可设计性，异常场景覆盖确认 |
| 相关服务负责人 | 确认跨服务接口契约无歧义 |

> **最小评审组合**：至少包含产品 + 开发 + QA 三方

## 门禁执行规范

- 评审会议时间不超过 2 小时（超时说明设计不成熟，打回重做）
- 评审结论写入设计文档注释或 Confluence 页面
- 通过后，接口契约冻结进入契约先行开发流程
- 评审中提出的修改项，修改完成后需相关方二次确认（可异步）
