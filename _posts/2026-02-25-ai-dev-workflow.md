# AI辅助研发工作流

## 定义

AI辅助研发工作流（AI-Augmented Development Workflow）是指在软件研发的每个关键节点引入 AI 工具，消除信息转换损耗，让开发者专注于真正需要业务判断的环节。

**核心理念：** AI 不替代开发，而是在每个"信息转换"节点担任翻译官和加速器。

## 信息转换损耗模型

传统研发的根本问题是**信息在各环节的转换损耗**：

```
PRD文本 ──人工解读──▶ 需求理解  [损耗：各人理解不一致]
需求理解 ──人工设计──▶ 接口文档  [损耗：设计与需求脱节]
接口文档 ──人工编码──▶ 代码骨架  [损耗：重复模板，耗时]
代码骨架 ──人工实现──▶ 业务逻辑  [损耗：经验不足，低质]
业务逻辑 ──人工审查──▶ CR评审   [损耗：经常跳过]
```

**AI 的介入价值：** 在每个"──人工──▶"节点注入 AI 能力，将损耗降到最低。

## 六层 AI 介入模型

### 层1：需求解析（PRD → 结构化需求）

**工具：** Claude Code

**输入：** PRD 文档（Markdown / Word / PDF 内容）

**输出：**
- 功能点编号清单
- 模块归属建议（按现有服务边界）
- 依赖的基础服务（权限/认证/工作流）
- 歧义问题列表（供产品澄清）

**核心价值：** 统一团队对需求的理解基准，歧义在编码前消除

---

### 层2：接口设计（需求 → 接口契约）

**工具：** Claude Code + Cursor

**输入：** 功能点清单 + 项目接口规范 + 现有接口示例

**输出（分两类）：**

| 产物 | 归属模块 | 用途 |
|------|---------|------|
| Feign Client（`I{Name}Client`） | `-api` 模块 | 跨服务调用契约，发布 Nexus 供其他服务依赖 |
| `*DTO`（数据传输对象） | `-api` 模块 | 跨服务调用的参数/返回对象 |
| Controller 方法签名 | `-svc` 模块 | 前后端交互的 REST API |
| `*VO`（视图对象） | `-svc` 模块 | 前端请求/响应数据结构 |
| 权限码命名建议 | 两者 | `@Permission` 注解的权限码 |

**核心价值：** 从"空白文档手写接口"→ "AI生成草稿，人工调整"；明确区分跨服务契约与前端 API

---

### 层3：骨架生成（接口 → 代码框架）

**工具：** 项目专属 Skill（如 `pm3-microservice-init`）+ Cursor

**输入：** 模块名称 + 接口文档

**输出：**
- 完整目录结构
- Controller / Service / Mapper / Entity / DTO / POM 全套文件
- 启动类、异常处理、常量定义

**核心价值：** 消除 60-70% 的重复模板代码编写

---

### 层4：业务实现（框架 → 可运行代码）

**工具：** Cursor（实时补全）+ Claude Code（复杂逻辑）

**典型场景：**
- 工作流定义生成（Activiti .bpmn20.xml）
- 复杂 SQL 查询生成（MyBatis XML）
- 幂等性/分布式事务代码模式
- 跨服务调用链路实现

**核心价值：** 开发者聚焦业务规则判断，AI 处理实现细节

---

### 层5：AI Code Review（代码 → 质量报告）

**工具：** Claude Code（PR 审查模式）

**检查维度：**
- 命名规范合规性
- 数据完整性（审计字段、软删除字段）
- 安全风险（SQL注入、越权、敏感数据）
- 代码质量（NPE、事务边界、异常处理）
- 业务逻辑正确性（对照需求描述）

**核心价值：** CR 从"经常被跳过"→ "AI自动执行，永不缺席"

---

### 层6：存量运维（代码 → 理解 + 修复）

**工具：** Claude Code

**典型场景：**

| 场景 | 输入 | 输出 |
|------|------|------|
| Bug 排查 | 异常日志 + 堆栈 + 相关代码 | 根因分析 + 修复建议 |
| 代码理解 | 复杂业务代码 | 中文注释 + 流程摘要 |
| 影响分析 | API变更描述 + Consumer代码 | 受影响调用点 + 修改方案 |
| 重构建议 | 存量代码 | 技术债清单 + 优先级建议 |

**核心价值：** 存量代码从"黑箱"→ "可理解、可追溯、可安全修改"

---

## 知识沉淀闭环

AI 辅助研发的最终价值在于**知识资产积累**，而不是一次性输出：

```
每次AI辅助任务完成
        │
        ▼
  5分钟整理归档
  (/parse-knowledge)
        │
  ┌─────┼──────┐
  ▼     ▼      ▼
有效  Bug    API变更
Prompt Pattern 案例
库     库      库
        │
        ▼
下次同类任务
AI 质量更高
```

**触发归档的时机：**
- PRD 解析完成，输出有效
- AI 生成代码通过评审（Prompt 有效）
- AI 帮助定位 Bug（根因 + 修复方案）
- -api 变更影响分析完成

## 工具选型原则

| 场景 | 首选工具 | 备选 | 不适合 |
|------|---------|------|--------|
| 需求分析、架构讨论 | Claude Code（对话） | - | Cursor |
| 实时代码补全 | Cursor | GitHub Copilot | Claude Code |
| 复杂逻辑生成 | Claude Code | Cursor | 纯手工 |
| 代码库全局理解 | Claude Code | - | Cursor |
| 单元测试生成 | Cursor | Claude Code | - |

## 实施成熟度阶梯

| 阶段 | 特征 | 目标指标 |
|------|------|---------|
| L0 不使用 | 纯手工开发 | 基线 |
| L1 辅助编写 | Cursor 实时补全 | 代码速度提升 30% |
| L2 骨架生成 | AI 生成 CRUD 骨架 | 新模块搭建时间 < 2小时 |
| L3 全流程辅助 | PRD→骨架→CR 全链路 | PRD到可测代码 < 1天 |
| L4 知识沉淀 | AI 辅助 + 知识库持续积累 | 同类需求复用率 > 60% |
