# 三轨并行架构

## 定义

三轨并行架构（Three-Track Parallel Architecture）是一种针对**大团队（>8人）微服务并行交付**的工作流组织模式，将研发活动拆分为三条同步推进的轨道：

- **Track A（平台工程轨）**：共享基础设施的 API 契约冻结与 Mock 发布
- **Track B（服务开发轨）**：各业务微服务的设计评审与并行编码
- **Track C（质量工程轨）**：CI/CD 门禁配置、契约测试框架搭建与集成验收

三轨的核心价值在于：**通过提前解耦依赖，消除服务间的等待链，实现真正意义上的并行交付**。

## 三轨职责与时序

```
时间轴 ──────────────────────────────────────────────────────►

Track A │ 共享服务  │──冻结API契约──►│发布Mock──►│维护&迭代
平台工程 │（Activiti） │ W1-2          │           │

Track B │ 服务1 │──设计评审──►│──────────编码──────────►│集成│
服务开发 │ 服务2 │──设计评审──►│──────────编码──────────►│集成│
（并行） │ 服务N │──设计评审──►│──────────编码──────────►│集成│
         │ 改造1 │──设计评审──►│──────────编码──────────►│集成│

Track C │ CI/CD卡口配置 │──►│契约测试框架─►│集成测试─►│性能│监控│
质量工程 │ W1           │   │ W2-8        │ W9-10    │    │
```

**关键时序原则：**
1. Track A 必须在 W1-2 内完成 API 契约冻结，这是所有 Track B 服务并行的前提
2. Track B 各服务之间互不等待，均通过 Track A 发布的 Mock 进行开发
3. Track C 贯穿全程，不是尾部阶段才介入的"测试轨道"

## 适用场景

| 条件 | 说明 |
|------|------|
| 团队规模 | >8人，存在多个服务负责人 |
| 架构形态 | 微服务/SOA，存在共享平台服务（工作流/消息/权限） |
| 交付周期 | 3个月内，需要高度并行以压缩关键路径 |
| 关键路径 | 存在多个服务依赖同一共享基础设施 |

**不适合的场景：**
- 小团队（<5人），单体应用，依赖少（无需三轨，直接串行开发效率更高）
- 共享服务不存在或依赖极少（无需 Track A）

## 三轨间的协调机制

### 依赖解耦：Mock First

Track B 服务不等待真实依赖，全部通过 Mock 开发：
```
Track A 发布 Mock（WireMock / Spring Cloud Contract Stub）
         │
Track B 服务1 ──► Mock调用 ──► 独立开发测试
Track B 服务2 ──► Mock调用 ──► 独立开发测试
```

### 契约变更管控

当 Track A 的 API 必须变更时：
1. 发出变更通知（Changelog 文件 + IM 群通知）
2. 更新契约文件（Groovy / YAML DSL）
3. Track B 所有 Consumer 更新 Stub 依赖
4. Track C 的契约测试自动验证一致性

### 每周同步节拍

建议采用固定节拍：
- **每周一**：Track A 同步 API 变更状态
- **每周三**：Track B 服务负责人交叉 CR
- **每周五**：Track C 运行全量契约测试，红绿状态共享

## 与其他模式的关系

| 模式 | 与三轨并行的关系 |
|------|-----------------|
| 设计评审门禁 | Track B 进入编码前的强制入口 |
| 契约先行开发 | Track A 与 Track B 之间的协作协议 |
| CI-CD质量卡口 | Track C 的核心工具 |
| Platform Engineering | Track A 的现代化命名（平台工程） |

## 小团队适配（<8人）

当团队规模较小时，可合并轨道：
- **两轨模式**：Track A（平台+质量）+ Track B（服务开发）
- 质量门禁依然存在，只是由同一批人维护
- 关键路径分析同样重要，避免 Mock First 原则被放弃
