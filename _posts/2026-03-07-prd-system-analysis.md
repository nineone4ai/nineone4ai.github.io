# PRD 系统分析

> 在 PRD 评审会议后，基于终版 PRD 进行的详细分析，输出完整的技术规格文档，指导开发实现

---

## 目的

- **技术翻译**：将 PRD 的业务语言转换为技术语言
- **完整规格**：输出字段、接口、模型、流程的完整定义
- **开发依据**：作为开发和测试的输入文档

---

## 核心内容

### 1. 功能模块分析

**字段清单**：
- 实体字段定义（类型、长度、必填、校验）
- 列表页字段（显示、筛选、排序）
- 表单页字段（必填、只读、联动）

**按钮清单**：
- 页面按钮定义（权限、前置条件、点击行为）
- 二次确认要求

**业务逻辑**：
- 业务规则（校验规则、计算规则）
- 状态流转（状态机）
- 数据联动（字段间、跨表）

**流程图**：
- 业务流程图（用户操作视角）
- 数据流程图（系统处理视角）
- 时序图（关键交互）

### 2. 数据模型设计

**ER 图**：概念模型，实体关系

**表结构**：物理模型，字段定义

**DDL**：可执行的建表语句

**索引设计**：查询优化

### 3. 对象模型设计

**类图**：Entity、Service、Controller 关系

**DTO**：请求/响应数据结构

**枚举**：状态、类型定义

### 4. API 接口设计

**接口清单**：所有接口的列表

**接口详情**：每个接口的请求/响应格式、错误码

### 5. 异常场景设计

**系统异常**：网络、超时、并发

**业务异常**：校验失败、权限不足、数据冲突

---

## 模板选择

| 模板 | 适用场景 | 核心内容 |
|------|---------|---------|
| 简化版 | 简单 CRUD、小功能优化 | 字段、按钮、流程、API、DDL |
| 完整版 | 常规业务模块 | 完整分析 + 对象模型 + 详细 API |
| 复杂工作流版 | 多角色协作、长流程 | 完整分析 + 状态机 + 时序图 + 通知机制 |

---

## 操作步骤

### Step 1：选择模板（5 分钟）

根据模块复杂度选择：
- 简单 CRUD → 简化版
- 常规业务 → 完整版
- 复杂工作流 → 复杂工作流版

### Step 2：填充文档（2-4 小时）

按照模板结构逐项填充：
1. 概述（15 分钟）
2. 功能模块分析（60-90 分钟/模块）
3. 数据模型设计（30-60 分钟）
4. 对象模型设计（30 分钟）
5. API 接口设计（30-60 分钟）
6. 异常场景设计（15 分钟）
7. 非功能性需求（15 分钟）
8. 依赖关系（10 分钟）

### Step 3：检查完整性（30 分钟）

对照 PRD 分析 CheckList 逐项检查：
- 字段完整性
- 按钮完整性
- 业务逻辑完整性
- API 接口完整性
- 异常场景完整性

### Step 4：评审和冻结（1 小时）

- 邀请产品、测试评审
- 根据评审意见修改
- 评审通过后冻结文档

## 输出物

### 《系统分析文档》

包含：
- 功能模块详细分析
- 数据模型（ER 图、表结构、DDL）
- 对象模型（类图、DTO、枚举）
- API 接口定义
- 异常场景设计

### 《开发任务清单》

包含：
- 后端开发任务（Entity、Mapper、Service、Controller）
- 前端开发任务（页面、API 封装、校验）
- 联调测试任务
- 工时评估

## 技巧

### 字段分析三遍法

1. 第一遍：对着原型图，列出所有显示字段
2. 第二遍：对着 PRD 逻辑描述，补充隐藏字段
3. 第三遍：用 CheckList 检查，确保不遗漏

### 流程图绘制技巧

1. 先画主流程（Happy Path）
2. 再画分支流程
3. 最后画异常流程
4. 用不同颜色区分不同角色/系统

### 接口设计原则

- RESTful 风格
- 统一的响应格式
- 清晰的错误码定义
- 详细的参数说明
